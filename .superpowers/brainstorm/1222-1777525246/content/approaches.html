3 Architectural Approaches for Pivot1

All three assume early-stage scale. Pick based on how much disruption you can tolerate.

A

Additive Migration Recommended

Add fields to existing tables. Extract jobs. Keep the session engine alive.

What changes
  • Add type + triggerCondition to OnboardingFlow
  • Keep table name — rename in code via Prisma alias
  • Extract cron jobs to apps/worker/ process
  • New GET /triggers/evaluate endpoint
Trade-offs
  • Zero breaking changes
  • Ship Phase 1 in 2 days
  • Live sessions unaffected
  • Table name stays messy short-term
Best for: You have live customers you can't break. Ship fast, rename later.
B

Schema Rename with Dual-Write

New AgentFlow table. Write to both during transition. Drop old table after.

What changes
  • New AgentFlow table with full schema
  • Dual-write: new code → AgentFlow, old → still works
  • Backfill script migrates existing flows
  • Drop OnboardingFlow after 2-week run
Trade-offs
  • Clean schema end-state
  • No planned downtime needed
  • 2x migration complexity
  • Dual-write bugs are subtle
Best for: You want a clean DB schema and have a week to do the migration carefully.
C

Event-Driven Core with Job Queue

Introduce BullMQ + Redis queues. All background work becomes async jobs.

What changes
  • Add BullMQ on top of Upstash Redis
  • Flow state changes emit events to queues
  • Workers consume: trigger-eval, email, churn
  • Schema rename (like B) + event sourcing
Trade-offs
  • Future-proof for growth-stage
  • Retries, dead-letter queues built-in
  • 3–4 weeks to implement
  • Overkill for early-stage
Best for: You're already at growth-stage. Not yet.
