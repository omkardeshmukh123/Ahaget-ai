Revised Architecture for Pivot1

You've already made the hardest shift — step-mode → react loop (goal mode). Here's what pivot1 actually needs now.

What the react loop already gives you for free
Any flow type
Goal mode doesn't care if the goal is "onboard" or "set up exports" — same loop works for adoption, retention, upsell
No step config needed
SaaS owner writes a goal in plain English. Agent figures out selectors at runtime. No pre-configured action maps.
Failure recovery built in
observe → retry → degrade_to_manual → escalate ladder already exists in agent.ts
What pivot1 actually needs to build on top of this
A

Thin Flow Envelope + Extracted Worker Recommended

Minimal schema additions. React loop stays untouched. Worker process handles all background jobs.

Schema (1 migration)
  • Add type to OnboardingFlow — default 'onboarding'
  • Add goalPrompt (text) — what the agent is trying to achieve
  • Add triggerCondition (JSON) — when to fire
  • OnboardingStep becomes optional (legacy only)
New: apps/worker/
  • Separate Node process, same Prisma client
  • Churn cron → worker
  • Inactivity trigger scan → worker
  • Proactive email dispatch → worker
Result: All 6 pivot1 phases can be built without touching the react loop. Worker handles everything async.
B

Full Schema Clean-up + Worker

Since you have no customers, do the clean rename now. Pay the cost once, get a clean DB forever.

Schema
  • Rename OnboardingFlow → AgentFlow
  • Drop OnboardingStep entirely (or archive)
  • New: type, goalPrompt, triggerCondition
  • Update all Prisma queries + routes in one PR
Trade-offs
  • Clean schema from day 1
  • No legacy OnboardingStep baggage
  • Half a day of search-and-replace
  • More surface area for bugs
Best for: No customers = no risk. This is actually a good time to do it cleanly.
C

Goal Orchestrator Layer (Advanced)

Wrap the react loop in a stateful orchestrator that manages multi-session continuity and cross-flow priority.

What it adds
  • FlowOrchestrator service: picks the highest-priority flow for a user
  • Pauses/resumes sessions across browser tabs
  • Handles "user qualifies for adoption AND retention" conflict
  • Turn history persisted in Redis (not just Postgres)
Trade-offs
  • Solves real multi-flow conflict problem
  • 3 weeks of work
  • You don't have the multi-flow conflict yet
Best for: Phase 3+ when you have adoption + retention flows both live. Not yet.
